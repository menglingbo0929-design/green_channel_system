# 高校绿色通道系统四人并行开发分工

## 一、分工原则

本项目采用前后端分离模式，技术栈为：

- 前端：Vue 3、Vue Router、Pinia、Axios、Element Plus、ECharts
- 后端：Spring Boot、MyBatis、MySQL
- 其他：JWT、文件上传、Excel 导入导出

四名成员按照业务模块进行纵向划分。每个人都负责自己模块涉及的：

1. 数据库表设计与 SQL；
2. Spring Boot 后端接口；
3. MyBatis 数据访问；
4. Vue 3 前端页面；
5. Axios 接口封装；
6. 前后端联调和测试。

四个人同时开始开发，不设置前后依赖式的等待流程。公共规范由四人提前统一，然后各自按约定并行完成模块。

---

### 4.1.4 绿色通道入口判断

根据以下条件判断学生是否可以进入申请页面：

- 当前是否存在开放批次；
- 当前时间是否在申请时间内；
- 学生是否属于允许申请的年级；
- 学生个人信息是否完善；
- 学生是否选择生源地贷款或校园地贷款；
- 学生是否已经提交过当前批次申请；
- 学生是否被学校允许代申请。

接口需要返回是否允许进入以及不能进入的具体原因。

- 新建申请草稿；

- 保存申请来源；

### 5.1.4 统一审核状态流转

建议使用以下申请状态：

```text
DRAFT                   草稿
COUNSELOR_PENDING       待辅导员审核
COUNSELOR_RETURNED      辅导员退回
COLLEGE_PENDING         待学院审核
COLLEGE_RETURNED        学院退回
SCHOOL_PENDING          待学校审核
SCHOOL_RETURNED         学校退回
REJECTED                审核不通过
APPROVED                学校审核通过
CONFIRM_PENDING         欠费待确认
COMPLETED               已完成
CANCELLED               已取消
```

正常流程：

```text
DRAFT
→ COUNSELOR_PENDING
→ COLLEGE_PENDING
→ SCHOOL_PENDING
→ APPROVED 或 CONFIRM_PENDING
→ COMPLETED
```

退回流程：

```text
COUNSELOR_PENDING
→ COUNSELOR_RETURNED
→ 学生修改
→ COUNSELOR_PENDING
```

```text
COLLEGE_PENDING
→ COLLEGE_RETURNED
→ 学生修改
→ COUNSELOR_PENDING
```

```text
SCHOOL_PENDING
→ SCHOOL_RETURNED
→ 学生修改
→ COUNSELOR_PENDING
```

每个审核接口必须检查当前状态，禁止绕过正常流程直接修改状态。

### 5.1.7 查看审核流程

返回内容包括：

- 当前审核节点；
- 历史审核记录；
- 审核人；
- 审核时间；
- 审核意见；
- 退回原因；
- 不通过原因。

针对线下申请、超时处理等情况，实现：

- 根据学号查找学生；
- 补录欠费申请；
- 补录礼包申请；
- 记录补录人和补录时间；
- 补录申请来源标记为 SUPPLEMENT；
- 补录后自动生成相应审核记录；
- 防止同一学生在同一批次重复补录。

### 6.1.5 统计功能

只对最终审核通过或已完成的数据进行统计。

统计内容包括：

- 申请总人数；
- 实报人数；
- 各学院申请人数；
- 各年级申请人数；
- 欠费项目人数；
- 欠费总金额；
- 欠费原因按人数统计；
- 欠费原因按金额统计；
- 爱心礼包物品申请数量；
- 历史批次数据。

### 6.1.6 统计筛选
