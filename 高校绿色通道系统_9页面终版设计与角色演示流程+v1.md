# 一、系统角色如何进入和使用

## 1. 统一登录入口

学生、辅导员、学院管理员、学校管理员都从同一个登录页面进入系统。

登录成功后，后端返回：

```text
JWT Token
用户 ID
姓名
角色
所属学院或学生 ID
数据权限范围
```

角色值统一为：

```text
STUDENT      学生
COUNSELOR    辅导员
COLLEGE      学院管理员
SCHOOL       学校管理员
```

前端根据角色生成不同菜单，后端根据角色和数据范围限制接口访问。

不能只通过前端隐藏菜单实现权限控制。即使用户手动输入没有权限的路由或调用接口，后端也必须拒绝。


## 3. 答辩时是否需要反复退出登录

正常系统中，一个登录会话只对应一个角色。

如果只在同一个浏览器窗口中演示，就需要：

```text
退出当前账号
→ 登录另一个角色账号
→ 进入对应角色页面
```

为了减少答辩中的重复操作，建议提前准备四个相互隔离的浏览器会话：

```text
Chrome 普通窗口：学校管理员
Edge：学院管理员
Firefox：辅导员
Chrome 无痕窗口：学生
```

也可以使用 Chrome 的不同用户配置文件。

不要在同一浏览器的普通标签页中同时登录多个角色，因为项目通常使用 `localStorage` 保存 Token，同一域名的标签页会共享登录状态。

答辩时只需要切换窗口，不需要反复退出和重新输入账号。

# 三、9 个页面的最终内容

## 页面 1：登录页

页面内容：

```text
系统名称
用户名
密码
登录按钮
```

登录成功后：

```text
读取用户角色
保存 Token 和用户信息
跳转角色默认页面
动态加载角色菜单
```

默认跳转：

```text
学生      → 学生申请中心
辅导员    → 辅导员审核页
学院      → 学院审核页
学校      → 学校业务处理页或统计看板
```

<div style="color:#C2410C">
<strong>【+v1 新增｜统一会话入口】</strong>
<ul>
<li>登录成功后必须继续调用当前用户能力，取得可信的用户 ID、姓名、角色和数据范围；前端不得自行拼接角色。</li>
<li>统一布局右上角显示当前用户、角色、修改密码和退出登录；修改密码复用“新增/编辑弹窗”。</li>
<li>Token 失效时清除本地会话并跳回登录页；无权限访问显示 403 状态，路由不存在显示 404 状态。403、404 属于系统状态，不计入 9 个业务页面。</li>
<li>顶部导航统一提供站内消息入口，显示未读数量并打开消息抽屉。</li>
</ul>
</div>

仅学生访问。

页面使用四个 Tab：

```text
个人信息

```

### 个人信息

只读：

```text

```

可编辑：

```text

资助认定
```

### 绿色通道

一个学生在同一批次只建立一条 `GREEN_CHANNEL` 主申请。

申请内部包括两个可选区域：

#### 欠费申请

```text
是否申请欠费
学费金额
住宿费金额
自动合计
欠费原因
补充说明
最高 8000 元提示
```

欠费原因必须使用固定枚举，而不是自由文本：

```text
FAMILY_FINANCIAL_DIFFICULTY  家庭经济困难
FAMILY_EMERGENCY             家庭突发变故
MAJOR_ILLNESS                重大疾病
DISASTER_ACCIDENT            灾害或事故
OTHER                        其他
```

该控件和“欠费原因统计”以 `arrears_application.arrears_reason_code` migration 已批准并合入为前提；此前页面只能展示补充说明，不能把补充说明用于统计分组。

#### 爱心礼包

```text
是否申请礼包
礼包物品卡片
尺寸
数量
剩余库存
必选标记
申请理由
```

页面还包括：

```text
附件上传
保存草稿
正式提交
```

### 生活补助与路费补助

两个 Tab 使用同一组件：


```text

```

### 入口资格

后端必须判断：

```text
存在开放批次
当前时间有效
年级符合
个人信息完善
已申请生源地贷款或拟申请校园地贷款
```

这里的“没有重复申请”统一解释为：当前学生、当前批次、当前申请类型不存在有效申请；草稿、审核中、已通过、待确认和已办结均属于有效申请，逻辑删除记录除外。

<div style="color:#C2410C">

<li>附件组件必须支持上传进度、列表、预览、删除未提交附件和重新上传；正式提交后按权限只读查看。</li>
<li>推荐卡片展示推荐的绿色通道或补助项目及推荐原因，不在前端自行计算学生画像。</li>
<li>绿色通道正式提交成功后，提示学生可继续申请生活补助或路费补助，并提供直接跳转入口。</li>
<li>草稿删除前显示将释放的预占资源；正式提交、重复点击和网络重试使用同一请求号，页面不得产生重复申请。</li>
</ul>

<li>顶部消息抽屉展示申请截止提醒、材料退回、审核结果、欠费待确认和线下办理通知；点击消息直接定位到对应申请详情，并记录已读状态。</li>

# 七、最终结论

本项目采用：

```text
一个登录入口
四种角色
九个主要页面
五类通用弹窗
一套统一申请和审核流程
```

角色登录后看到的菜单、数据和操作不同。

正常使用时，切换角色需要退出并重新登录；答辩演示时使用四个独立浏览器会话，可以直接切换角色视图，不必反复退出登录。

该页面设计已经覆盖 V2 的全部核心业务，同时将页面数量和演示复杂度控制在课程项目可接受范围内。
