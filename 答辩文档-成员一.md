## 一、项目背景

高校绿色通道系统是一个面向高校新生资助工作的数字化平台。传统资助流程依赖线下纸质申请、逐级签字审核，效率低且学生无法提前办理。本系统将绿色通道（学费缓缴）、困难补助从线下搬到线上，让新生从收到录取通知书起即可在家办理资助申请。

### 四角色协作模式

| 角色 | 职责 | 分支 |
|------|------|------|
| 成员一（本人） | 基础用户、学生管理、组织结构、批次管理、认证授权 | `feature/base-user-batch` |
| 成员二 | 申请配置、欠费项目、礼包、补助额度 | `feature/application-config` |
| 成员三 | 三级审核流程、状态机、消息通知 | `feature/approval-flow` |
| 成员四 | 欠费确认、学校代申请、补录、统计报表 | `feature/confirmation-statistics` |

## 四、后端架构

### 4.1 安全认证

```
登录请求 → SecurityConfig 白名单放行
         → UserController.login()
         → BCrypt 验密 + 查角色 + 查学生信息
         → JWT 签发（含 userId, roles, studentId, collegeId）
         → JwtAuthenticationFilter 每次请求解析
         → CurrentUserProvider 从 SecurityContext 获取当前用户
```

**特色功能：**
- 密码登录 + 短信验证码登录双模式
- 验证码 60s 频率限制、5 分钟有效期、一次性消费
- 登录历史记录（localStorage 持久化，退出不清空）
- 停用用户自动拒绝登录

### 4.2 跨模块服务（6 个）

向成员二、三、四提供标准化后端能力：

| 服务 | 调用方 | 功能 |
|------|--------|------|
| `CurrentUserProvider` | 全员 | 从 JWT 获取当前用户 ID/角色/学生ID/学院ID |
| `StudentProfileQueryService` | 成员二 | 获取学生画像（含组织名称和贷款信息） |
| `StudentScopeService` | 成员二、三 | 校验辅导员管辖关系、学院归属、学生本人 |
| `BatchQueryService` | 全员 | 查询开放批次、校验年级资格、批次列表 |
| `OrganizationQueryService` | 成员二 | 学院/年级/专业下拉选项 |
| `PolicyRuleQueryService` | 成员二 | 当前批次政策规则提示 |

### 4.3 Port 实现（成员四/二要求的接口）

| Port | 功能 |
|------|------|
| `SchoolProxyStudentQueryPort` | 按学号/ID/批量查学生快照（含组织 ID） |
| `StatisticsAccessPort` | 统计权限校验（SCHOOL 角色） |

## 五、功能模块

### 5.1 用户管理

- 用户 CRUD + 多角色分配（一个用户可同时拥有多个角色）
- 启用/停用切换（停用用户仍在列表中显示，但无法登录）
- 管理员、学校管理员、学院管理员、辅导员、学生五种演示账号

### 5.2 组织结构管理

- 学院 / 专业 / 年级 / 班级四级 CRUD
- 专业关联学院、班级关联学院+专业+年级
- 启用/停用独立控制

### 5.3 学生管理

- 学生 CRUD + 多条件筛选
- Excel 批量导入：自动创建 `sys_user` 登录账号 + 绑定 STUDENT 角色
- 模板下载：含示例数据的标准导入模板
- 导入结果反馈：成功数、跳过数、错误详情弹窗

### 5.4 绿色通道批次管理

- 批次 CRUD：编号、名称、申请起止时间、学院截止时间
- 适用年级设置
- 启用/停用 + 状态管理（DRAFT / OPEN / CLOSED）
- 成员二/三通过 `BatchQueryService` 查询批次信息

### 5.5 辅导员-学生关联

- 辅导员分配/解除学生
- 批量分配
- 查询辅导员管辖学生列表 / 学生对应辅导员列表

### 5.6 验证码登录

- 手机号 + 6 位验证码登录
- 查 `student.phone` → 查 `sys_user` → 签 JWT
- 60s 发送频率限制，5 分钟有效期

## 九、关键技术决策

1. **JWT 无状态认证**：Token 承载 userId + roles + studentId + collegeId，不依赖 Session
2. **多角色模型**：用户-角色多对多，支持学生兼任辅导员等灵活场景
3. **逻辑删除统一**：`deleted BIGINT DEFAULT 0`，与系统其他模块一致
4. **组织 ID 冗余**：`class_info` 同时存 `college_id` + `major_id` + `grade_id`，避免多表 JOIN
5. **验证码内存存储**：开发阶段使用 ConcurrentHashMap，生产可切换 Redis
6. **Excel 导入原子操作**：每行独立事务，学号重复自动跳过

## 十、项目亮点

- **完整的基础设施**：从 JWT 认证到前端 Layout，四人团队的统一底座由成员一搭建
- **6 个跨模块 Service**：清晰定义接口边界，其他成员不直接查表
- **15 张表完整设计**：覆盖用户、角色、组织、学生、批次、标签、政策规则
- **Excel 批量导入**：一条龙自动创建账号 + 绑定角色 + 关联学生，导入结果可视化
- **双模式登录**：密码 + 验证码，登录历史记录提升用户体验
- **严格协作规范**：所有修改文档同步、表所有权清晰、跨模块写入必须走 Service
